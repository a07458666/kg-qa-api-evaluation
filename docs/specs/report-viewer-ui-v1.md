# Report Viewer UI v1

## Goal

Build a static report viewer for `kg-qa-api-evaluation` that:
- renders KG results as interactive graphs with D3.js
- renders QA answers as rich Markdown content
- reads report JSON files generated by the evaluator
- works without a backend server

This viewer should remain agent-agnostic and portable.

## Why this approach

The current project already separates:
- evaluation data
- presentation metadata

That makes a static UI feasible before the final backend response schema is fixed.

The viewer only needs the report JSON contract, especially:
- top-level `presentation`
- per-case `presentation.graph`
- per-case `presentation.qa`

## Inputs

### Required
- one report JSON file conforming to `schemas/report.schema.json`

### Optional
- baseline report JSON for side-by-side regression views
- theme override config in the future

## Output

A static HTML bundle with:
- `index.html`
- `styles.css`
- `app.js`
- optional sample report JSON

No framework is required for v1.

## Technology choices

### Core UI
- static HTML
- CSS variables for theming
- vanilla JavaScript modules

### Graph rendering
- D3.js
- default layout: force-directed graph

Reason:
- already aligned with the report presentation schema
- best fit for graph exploration and custom legend/status styling

### Markdown rendering
- `markdown-it` or `marked`
- GitHub Flavored Markdown support required

Must support:
- tables
- bold / italic
- lists
- blockquotes
- inline code / fenced code blocks

### Sanitization
- DOMPurify recommended for rendered Markdown

Reason:
- report files may eventually contain user-facing text, citations, or model output

## Visual direction

Based on the UI/UX skill guidance, use an analytics-dashboard style:
- primary style: Data-Dense + Minimalism
- optional dark mode support
- emphasis on readability and information hierarchy
- tables must be wrapped in horizontal scroll for mobile safety

## Information architecture

### 1. Overview page
Show:
- project name
- environment
- run timestamp
- mode
- pass/fail summary cards
- latency summary
- schema failure count
- regression status if present

### 2. KG results tab
For each KG case:
- case header
- pass/fail badge
- failure reasons
- metrics summary
- D3 graph canvas
- legend
- raw normalized entity/relation tables

### 3. QA results tab
For each QA case:
- case header
- pass/fail badge
- failure reasons
- metrics summary
- rendered Markdown answer
- citations section
- raw normalized output accordion

### 4. Regression tab
If `regression_summary` exists, show:
- gating result
- metrics delta cards
- worsened cases list
- improved cases list

## D3.js behavior plan

### Default graph behavior
- use force simulation
- draggable nodes
- zoom and pan enabled
- hover tooltip for node/link metadata
- click node to highlight neighbors

### Styling semantics
Node color:
- by `group`

Node outline or glow:
- by `status`
  - matched
  - missing
  - unexpected
  - forbidden

Edge color:
- by `status`

### Accessibility fallback
Because network graphs are not sufficient alone, always also render:
- entity table
- relation table

This is consistent with the external UI skill guidance.

## Markdown rendering plan

### Source priority
1. `presentation.qa.answer_markdown`
2. fallback to `normalized_output.answer`

### Rendering rules
- use GFM
- sanitize HTML
- preserve tables
- wrap wide tables in scroll container
- syntax highlight can be deferred to v2

### Citation rendering
If `presentation.qa.citations` exists:
- render each citation as a card or compact table row

Otherwise fallback to:
- `normalized_output.citations`

## Filtering and interaction

### Search

## Recommendation from UI/UX skill applied here

Applied guidance:
- analytics-dashboard layout is the right product pattern
- D3.js is appropriate for relationship/network data
- provide non-graph fallback tables for accessibility
- ensure text readability and mobile-safe table overflow handling
- prefer structured information hierarchy over decorative UI
